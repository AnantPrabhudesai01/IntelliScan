System Overview

The IntelliScan class diagram shows the core data entities and their relationships. The system includes user management (User, Workspace, Team), contact management (Contact, Card), marketing capabilities (Campaign), integrations (CRMIntegration), AI processing (AIModel), and comprehensive tracking (Analytics, AuditLog).

Class Relationships & Cardinality

One-to-Many (1 → *) Relationships

User → Contact
One user creates many contacts. Referenced as: User.id → Contact.userId
User → Workspace
One user owns/belongs to many workspaces. Cardinality: (1) User [owns] (*) Workspaces
Workspace → Team
One workspace has many teams. Referenced as: Workspace.id → Team.workspaceId
Workspace → Campaign
One workspace has many campaigns. Referenced as: Workspace.id → Campaign.workspaceId
Workspace → Analytics
One workspace generates many metrics. Referenced as: Workspace.id → Analytics.workspaceId
Team → Contact
One team is assigned many contacts. Through: ContactAssignment junction table
Contact → Card
One contact may have multiple card images (front, back). Cardinality: (1) Contact has (1..*) Cards
User → Campaign
One user creates many campaigns. Referenced as: User.id → Campaign.createdBy
Workspace → CRMIntegration
One workspace has multiple CRM integrations. Cardinality: (1) Workspace has (0..* ) CRMIntegrations

One-to-One (1 → 1) Relationships

Contact → Card (Primary)
Each contact may have a primary/featured card. Referenced as: Contact.primaryCardId → Card.id (nullable)
User → Workspace (Default)
Each user may have a default workspace. Referenced as: User.defaultWorkspaceId → Workspace.id (nullable)

Many-to-Many (*:*) Relationships

User ↔ Workspace
Junction Table: WorkspaceMember (workspaceId, userId, role, joinedAt)
User ↔ Team
Junction Table: TeamMember (teamId, userId, role, joinedAt)
Contact ↔ Campaign
Junction Table: CampaignRecipient (campaignId, contactId, status, sentAt)
Workspace ↔ AIModel
Junction Table: WorkspaceAIModel (workspaceId, aiModelId, isPrimary, enabledAt)

Data Constraints & Validations
Constraint Type Details
Primary Keys All IDs are UUIDs (auto-generated), immutable after creation
Unique Constraints User.email (system-wide unique), Contact.email + Contact.workspaceId (unique per workspace), Workspace.domain
Foreign Keys All foreign key references include CASCADE delete for child records, SET NULL for audit records
Email Validation Must match standard email regex pattern if provided. Unique per system and workspace
Phone Validation Must be valid E.164 format if provided. International format with + prefix
URL Validation Contact.website and Card.imageUrl must be valid URLs starting with http:// or https://
Storage Constraints StorageLimit ≥ StorageUsed always enforced. ContactCountLimit ≥ ContactCountCurrent
Confidence Scores Must be Float between 0.0 and 1.0 inclusive. Used for quality assessment
Soft Deletes Contacts and Users support soft delete (deletedAt timestamp) rather than hard delete for audit trail
